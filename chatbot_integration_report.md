# 🤖 تقرير: آلية ربط روبوتات المحادثة (Chatbots) بنظام منصة وداد

هذا التقرير يوضح بالتفصيل الطريقة التي تم بها دمج وربط روبوتات المحادثة الذكية بنظام منصة "وداد"، وكيفية تواصل واجهة المستخدم (Front-End) مع الخادم (Back-End) ونماذج الذكاء الاصطناعي الخارجية المستضافة على Hugging Face.

---

## 1. نظرة عامة على هيكلية الربط (Architecture)

يعتمد الربط على معمارية متعددة الطبقات تضمن تقديم ردود سريعة وإدارة ذكية للطلبات. النظام يتكون من ثلاث محطات رئيسية:

1. **واجهة المستخدم (React 19 + TypeScript):** الوجهة التي يتفاعل معها الزائر أو المريض.
2. **الخادم (Laravel 12):** الوسيط الذي يدير الطلبات، ويعالج البيانات، ويتحكم في الطوابير (Queues)، ويطبق طبقة الحماية (PII Sanitization).
3. **نماذج الذكاء الاصطناعي (Hugging Face Gradio Spaces):** أربعة روبوتات متخصصة (عام، ما قبل الزواج، حمل، أمومة) مجهزة لاستقبال البيانات عبر `Gradio API`.

---

## 2. آلية الربط مع نماذج الذكاء الاصطناعي (Hugging Face - Gradio API)

الربط الفعلي بين الخادم (Laravel) و(Hugging Face) لا يتم عبر API تقليدي للدردشة، بل يتم باستخدام واجهة برمجة التطبيقات الخاصة بـ **Gradio (Server-Sent Events - SSE)**. وتستند هذه الآلية إلى الخطوات التالية:

* **بدء الطلب (POST):** يرسل الخادم طلبًا إلى مسار `/gradio_api/call/chatbot_fn`.
* **استلام معرف الحدث (Event ID):** يستجيب Gradio بمعرف فريد للحدث `event_id`.
* **الاستماع للتدفق (GET + SSE):** يقوم الخادم بإجراء طلب GET إلى `/gradio_api/call/chatbot_fn/{event_id}` للاستماع إلى البيانات التي يتم إرسالها بشكل متدفق (Stream).
* **تحليل البيانات (Parsing):** الخادم مجهز بوظيفة `parseSSEResponse()`، التي تقرأ البيانات المتدفقة، تستبعد النبضات الفارغة (Heartbeats)، وتستخرج النص النهائي الخاص برد الروبوت حيثما يكون `role="assistant"`.

---

## 3. دورة حياة الرسالة (Message Flow)

تختلف طريقة المعالجة بناءً على نوع المستخدم:

### أ. مسار المريض (Async + Polling)
لضمان عدم توقف النظام أو متصفح المستخدم بسبب تأخر رد الذكاء الاصطناعي (Cold Starts)، تم تنفيذ الآلية التالية:
1. **الإرسال:** يقوم المريض بإرسال رسالة من الواجهة (React).
2. **الحفظ المبدئي:** يستقبل (Laravel) الرسالة، يحفظها في قاعدة البيانات `ai_chat_messages` بحالة `pending`، ويعيد `message_id` فوراً للواجهة.
3. **واجهة المستخدم (Polling):** تبدأ واجهة React في عمل طلبات قصيرة (Polling) كل ثانيتين (بحد أقصى 60 ثانية) للسؤال عن رد البوت عبر مسار `/patient/chatbot/messages/{id}/status`.
4. **الطابور (Queue):** في الخلفية، تُرسل المهمة `ProcessChatbotMessageJob` إلى طابور المعالجة `Queue`.
5. **جلب الرد:** يتواصل الطابور مع Hugging Face للحصول على الرد، يحدّث حالة الرسالة في قاعدة البيانات لتصبح `ready`، ويحفظ الرد.
6. **عرض الرد:** بمجرد أن يجد طلب الـ Polling أن الحالة أصبحت `ready`، تقوم الواجهة بعرض الرسالة أخيرًا وتُصدر تنبيهًا صوتيًا.

### ب. مسار الزائر (Sync)
للزوار، يتم إرسال الطلب عبر مسار `/chatbot/public/message` ليُعالج بشكل فوري ومتزامن (Synchronous). نظرًا لطبيعة أسئلة الزوار السريعة والعامة، ينتظر الخادم رد نموذج Hugging Face ويعيده مباشرة في نفس الـ Request، مع الاعتماد بشكل كبير على "التخزين المؤقت" (Cache) لتسريع الاستجابة.

---

## 4. طبقة التخزين المؤقت الداعمة (Redis Caching)

لتقليل التكلفة، وتقليل وقت الاستجابة للمستخدم، وتفادي الوصول لحد الاستخدام الأقصى على Hugging Face، يطبق الخادم سياسة تخزين مؤقت (Caching) فعّالة:

* عند استلام الخادم أي رسالة، يتحقق من **Redis Cache** لمعرفة إذا كان قد تلقى سؤالًا مشابهًا لهذا الروبوت من قبل.
* أذا وُجد الرد في الـ Cache، يتم اقتطاع دورة الاتصال بـ Hugging Face وإرجاع الرد مباشرة.
* إذا لم يُوجد، يتم الاتصال بـ Hugging Face، ثم يُخزن الرد الجديد في Redis لمدة **24 ساعة**.

---

## 5. حماية الخصوصية (PII Sanitization & Emergency)

نظرًا للتعامل مع بيانات صحية حساسة، تم ربط الروبوت وتضمين آليات حماية ضمن طبقة الاتصال:

* **تنقيح البيانات (Sanitization):** قبل إرسال نص رسالة المستخدم إلى Hugging Face، يمر المُدخل عبر وظيفة تمسح البيانات الشخصية الحسّاسة (PII) مثل الإيميلات، أرقام الهواتف، والأرقام القومية، وتستبدلها بوسوم مثل `[REDACTED_PHONE]`. بذلك لا تذهب بيانات شخصية إلى النماذج الخارجية.
* **رصد الطوارئ (Emergency Detection):** واجهة المستخدم (Front-End) تحتوي على قاموس للكلمات الحرجة. بمجرد اكتشاف النص المرسل لكلمات تندرج تحت بند الخطر الصحي، يظهر إشعار طوارئ (`EmergencyCard`) تلقائيًا بضرورة التدخل البشري الفوري.

---

## 6. ملخص أدوات الربط في الكود

- **Front-End:** تعتمد على Custom Hooks وهي: `usePublicChatbot`, `useAuthChatbot`, `useWidgetChatbot`. بالإضافة إلى Axios Instance بـ Timeout طويل يبلغ (65 ثانية).
- **Back-End:** تعتمد على Service class مركزي `ChatbotService.php`، ومتحكمات (Controllers) تُفرق بين مسارات المرضى والزوار والإدارة، بجانب Jobs لمعالجة الطوابير في الخلفية `ProcessChatbotMessageJob`.
